E-Commerce · Live Auction · KSA

Soldout - Customer & Vendor Platform

Dual-app KSA marketplace combining standard e-commerce with live auction mechanics. Sole engineer on both the buyer-facing marketplace and the merchant management application.

Android iOS Vendor iOS
Problem · Solution

Problem. A live auction marketplace can't share state across features. A user actively bidding cannot be interrupted by a cart update, a push event, or a navigation action - that path leads to race conditions, stale bid UIs, and lost conversions. Saudi buyers also expect native Apple Pay and BNPL at checkout; anything less is friction.

Solution. Three isolated Cubit domains - CartCubit, AuctionCubit, MainCubit - each with its own lifecycle. Socket.IO processes real-time bid events without touching the UI state tree. A unified checkout abstraction keeps the UI provider-agnostic: Apple Pay, Tabby, and Tamara plug into the same interface.

Key decision: post-auction hand-off from AuctionCubit to CartCubit runs through a unidirectional event bridge instead of direct Cubit communication - both modules stay independently testable.
My Role · Impact

Role. Sole Flutter engineer on both the buyer marketplace and the merchant management app. Owned architecture, implementation, App Store + Play Store release, and ongoing maintenance.

Impact. 2 apps live in KSA across iOS + Android. Zero bid-state race conditions observed after launch. Checkout supports 3 payment rails natively (Apple Pay, Tabby, Tamara). Promotional coins and referral wallet modeled as immutable value objects - transaction history is reconstructable from state snapshots.

Stack discipline. Firebase Analytics, Remote Config, Crashlytics, and AppsFlyer registered in GetIt at startup and injected where needed - never referenced directly from UI code.

Real Estate · UAE · Government Auth

AlMarwan - Property Management Platform

Full-featured property management system for UAE investors - owned units, payment workflows, cheque management, document handling - all behind UAE PASS government authentication.

App Store
Problem · Solution

Problem. UAE investors manage owned units, payment workflows, cheque lifecycles, and document uploads - all behind UAE PASS, a government-backed identity system with strict OAuth 2.0 flows and certificate chains. Any credential touching Presentation is a compliance failure. A stale render on a cleared cheque has real financial consequences.

Solution. Authentication sealed as a Domain module - the app shell only sees authenticated or unauthenticated. Tokens refresh inside Secure Storage; no credential ever reaches UI code. Financial workflows use pessimistic state updates - UI reflects a completed transaction only after explicit server confirmation.

Real-time design: cheques cleared by the backend surface immediately in the investor dashboard. Socket.IO listeners are scoped to the active property context, with auto-reconnect and optimistic UI rollback on failure.
My Role · Impact

Role. Flutter engineer responsible for the end-to-end mobile product - UAE PASS integration, real-time cheque/payment flows, document handling, and App Store release.

Impact. Live on the App Store. Government-grade auth flow with zero credential leakage to Presentation. Real-time financial state delivered via scoped socket contexts. Typed document pipeline with client-side MIME + size validation - failed uploads never corrupt payment state (failures are scoped).

Multi-Brand · Single Codebase · Flavorizr

Markatty - Multi-Brand Storefront System

One Flutter codebase powering multiple distinctly-branded customer apps and a merchant platform. Eliminated duplicate codebases entirely through a Flavorizr-based multi-flavor architecture.

iOS Store Vendor Android
Problem · Solution

Problem. One storefront platform needed to power multiple distinctly-branded customer apps plus a merchant console - different icons, splash screens, bundle IDs, Firebase projects, and env vars per brand. The typical fix - duplicating codebases - is a maintenance trap that scales linearly with brand count.

Solution. Flavorizr-driven multi-flavor architecture. Each flavor declares its own complete configuration. GetIt injects the correct config object at startup. The flavor boundary sits at config-injection - zero conditional logic inside business or UI code.

Result: adding a new brand is a configuration task, not a code task. The core app is brand-agnostic by design - the same binary logic ships as every storefront.
My Role · Impact

Role. Sole architect of the multi-flavor system. Built Markatty Vendor end-to-end (order management, analytics, auth, geolocator features, secure merchant storage) and shipped the shared customer-app core.

Impact. Multiple branded storefronts live from one codebase - duplicate codebases eliminated entirely. Hybrid state strategy applied deliberately: BLoC for complex domains (cart, checkout, orders), Provider for lightweight local state. BLoC everywhere adds overhead on simple flows; Provider on checkout sacrifices testability - one tool for every job is a bad default.

Fashion E-Commerce · 3D Viewer

La Mode - Fashion Platform with 3D Product Preview

End-to-end fashion shopping experience built from scratch - OTP auth, interactive 3D product visualisation, complete checkout flows, full Arabic/English RTL localisation.

GitHub
Problem · Solution

Problem. A fashion app with interactive 3D product previews (GLTF/GLB via model_viewer_plus) has a memory problem: 3D assets are heavy, and browsing multiple products without an explicit disposal strategy exhausts memory progressively until the app chokes.

Solution. Lazy-load with explicit disposal. The 3D model initializes only when the product detail screen is active and is disposed immediately on pop. Combined with Shimmer placeholders, the experience feels instant without holding assets between screens. RTL handled once at the app root via MainCubit - no screen-level locale logic.

Persistence: Hive provides an offline-capable product cache - browsed items survive app restart. Clean Architecture with BLoC for core flows (auth, cart, orders, promo codes); Provider for lightweight local state.
My Role · Impact

Role. Personal open-source project. Full ownership - architecture, 3D integration, state management, RTL localization, offline cache, and GitHub release.

Impact. Production-grade 3D product viewer with memory-safe lifecycle. Full Arabic + English RTL from day one. Offline browsing via Hive. Codebase serves as a reference implementation of Clean Architecture in a consumer-facing Flutter app.

Engineering Philosophy

How I Think

Principles applied consistently across every project - not preferences that shift per client.

01

Architecture First

Every project uses Clean Architecture with three explicit layers. Business logic belongs in Domain - always. The Presentation layer contains nothing a stakeholder would care about. This determines whether the app is maintainable in year two or needs a rewrite.

02

State Management with Intent

BLoC/Cubit for complex domains - explicit state machines make bugs visible and tests straightforward. Provider for simple local state. The mistake is one tool everywhere: BLoC on a form toggle buys nothing; Provider on a checkout loses auditability.

03

Real-Time as a State Machine

Socket.IO events are state transitions, not data updates. A bid event may invalidate the cart, reset a countdown, and emit a notification simultaneously. Each effect is handled in Domain as a side-effect chain - not in the widget. This prevents UI thrashing.

04

Type-Safe API Layers

Dio + Retrofit + JSON code generation makes the API contract compile-time safe. Manual JSON parsing is a class of bugs that only surface in production when the backend changes a field name. Code generation turns runtime bugs into compile-time errors.

05

GCC Market Expertise

Shipping for GCC is not an Arabic translation file. It requires UAE PASS flows, BNPL providers (Tabby, Tamara), and platform compliance for Saudi/UAE stores. Arabic strings can be 1.3–1.6× longer than English - RTL layouts must be designed for it from day one.

06

End-to-End Ownership

I don't hand off to DevOps for deployment. Every project: CI/CD, App Store Connect, Play Store staged rollouts, code signing, release versioning. Architectural decisions interact with deployment constraints - knowing both means I don't design systems that can't be shipped.
